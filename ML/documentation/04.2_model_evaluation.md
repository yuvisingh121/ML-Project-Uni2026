### Task 4 – Model Evaluation
Valutazione del classificatore Decision Tree sul dataset esteso

## 0. Obiettivo del Task
In questo task viene valutato il classificatore Decision Tree progettato manualmente nel Task 2, applicandolo al dataset esteso `training.csv` composto da 50.000 osservazioni.
Lo scopo è verificare la capacità del modello di generalizzare quando viene utilizzato su un campione molto più ampio rispetto al dataset manuale usato per costruire le regole dell'albero, e tentare di ottimizzarne le prestazioni.

## 1. Preparazione del dataset
Per garantire coerenza con il modello costruito nel Task 2, viene applicata al dataset `training.csv` la stessa trasformazione:
- discretizzazione della variabile BMI nelle categorie Normal (BMI < 25), Overweight (25 ≤ BMI < 30), Obese (BMI ≥ 30)

L'albero utilizza due sole feature: BMI_Category (ottenuta dalla discretizzazione) e GenHlth. Quest'ultima è già espressa come variabile ordinale (da 1 = salute ottima a 5 = salute pessima), quindi non richiede alcuna trasformazione e può essere utilizzata direttamente dal classificatore.

## 2. Applicazione del classificatore
L'albero costruito nel Task 2 viene tradotto nella stessa funzione Python e applicato a tutte le osservazioni del dataset. Le regole di decisione sono:

1. SE BMI è Obese → diabetico (1)
2. SE BMI è Overweight E GenHlth ∈ {2, 4} → diabetico (1)
3. SE BMI è Overweight E GenHlth ∈ {1, 3} → non diabetico (0)
4. SE BMI è Normal → non diabetico (0)

Per ogni osservazione l'albero viene percorso dalla radice fino a una foglia, e la classe corrispondente viene assegnata. Il risultato della predizione viene salvato nella colonna Predicted.

## 3. Tabella dei risultati
Per ogni riga del dataset vengono confrontati:
- valore reale: Diabetes_binary
- valore predetto: Predicted

Questa tabella costituisce la base per la costruzione della confusion matrix e delle metriche di valutazione.

## 4. Confusion Matrix
La matrice di confusione riassume il comportamento del modello distinguendo:
- True Negative (TN): non diabetici correttamente classificati
- False Positive (FP): non diabetici classificati come diabetici
- False Negative (FN): diabetici non rilevati
- True Positive (TP): diabetici correttamente classificati

La distribuzione di questi valori permette di comprendere quali errori il modello commette più frequentemente.

## 5. Metriche di valutazione
A partire dalla confusion matrix vengono calcolate le metriche standard:
- Accuracy = (TP + TN) / (TP + TN + FP + FN)
- Precision = TP / (TP + FP)
- Recall = TP / (TP + FN)
- F1-Score = 2 * (Precision * Recall) / (Precision + Recall)

Applicando l'albero originale al dataset `training.csv` si ottengono i seguenti valori:
- Accuracy = 53.40%
- Precision = 19.01%
- Recall = 71.91%
- F1-Score = 30.07%

La confusion matrix corrispondente è: TN = 21.690, FP = 21.343, FN = 1.957, TP = 5.010. Si osserva immediatamente un numero molto elevato di falsi positivi.

## 6. Interpretazione dei risultati
Il modello Decision Tree costruito manualmente mostra limiti evidenti quando applicato a un dataset molto più grande:
- le regole erano state ricavate da soli 14 campioni, un insieme troppo piccolo per catturare i pattern reali presenti in decine di migliaia di osservazioni
- la regola "Obese → sempre diabetico" non è realistica: nel dataset completo solo una minoranza dei soggetti obesi è effettivamente diabetica, e questo genera un numero molto elevato di falsi positivi
- la regola basata sui valori pari di GenHlth ({2, 4}) era una coincidenza del piccolo campione e non riflette una relazione reale: nei dati veri la probabilità di diabete cresce in modo monotono all'aumentare di GenHlth

L'elevato numero di falsi positivi spiega la bassa precision e la bassa accuracy. La recall resta invece relativamente alta, perché un modello che predice "diabetico" molto spesso tende comunque a intercettare gran parte dei veri positivi.

## 6.1 Ottimizzazione del classificatore
Per migliorare le prestazioni, le regole dell'albero sono state ridefinite sulla base dei pattern osservati in un campione stratificato di 1000 osservazioni estratto dal dataset di training. L'analisi di tale campione ha evidenziato due andamenti chiari:
- la percentuale di diabetici cresce in modo monotono con il BMI (Normal < Overweight < Obese), ma anche tra gli obesi i diabetici restano una minoranza
- la percentuale di diabetici cresce in modo monotono con GenHlth (dal ~2% per GenHlth = 1 fino a ~32% per GenHlth = 5)

Sulla base di questi pattern, l'albero è stato riscritto combinando entrambe le feature con una soglia crescente su GenHlth, più bassa per le categorie di BMI a rischio maggiore:
- Obese → diabetico se GenHlth ≥ 3
- Overweight → diabetico se GenHlth ≥ 4
- Normal → diabetico se GenHlth ≥ 5

Applicando l'albero ottimizzato al dataset `training.csv` si ottengono i seguenti valori:
- Accuracy = 75.00%
- Precision = 30.14%
- Recall = 60.24%
- F1-Score = 40.18%

La confusion matrix corrispondente è: TN = 33.305, FP = 9.728, FN = 2.770, TP = 4.197.

Il confronto tra il modello originale e quello ottimizzato evidenzia un netto miglioramento di accuracy, precision e F1-Score, ottenuto soprattutto grazie alla drastica riduzione dei falsi positivi (da 21.343 a 9.728). La recall diminuisce leggermente: il modello ottimizzato è più conservativo, individua qualche diabetico in meno ma con maggiore affidabilità. Si tratta del classico compromesso tra precision e recall, in cui il guadagno complessivo (misurato dall'F1-Score) risulta positivo.

## 7. Conclusioni
Il classificatore Decision Tree manuale costituisce un valido esercizio didattico per comprendere il funzionamento del modello. Le regole derivate da soli 14 campioni non generalizzano sul dataset reale, ma ricalibrandole sui pattern osservati in un campione rappresentativo è stato possibile recuperare in misura significativa le prestazioni, pur restando nei limiti di un modello a regole semplici e interpretabili.

Anche dopo l'ottimizzazione, le prestazioni restano modeste in termini assoluti: un albero a due sole feature non può competere con un classificatore in grado di sfruttare l'intero insieme delle variabili disponibili. Il passo successivo consiste nel confrontare queste prestazioni con modelli implementati tramite librerie standard come Scikit-Learn.
